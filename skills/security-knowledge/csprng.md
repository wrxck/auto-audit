# Cryptographically secure randomness

Non-cryptographic random functions are seeded from low-entropy sources (system time, process state) and produce output that is **predictable** to anyone who can guess or observe a small number of prior outputs. Using them for security-sensitive values — session tokens, password reset tokens, CSRF tokens, API keys, cryptographic nonces, salts, IDs that double as access controls — lets an attacker reconstruct the PRNG state, predict future outputs, or recover past ones.

This is not a theoretical class. Real-world examples: a mass session-hijack vulnerability when a popular framework used `Math.random()` for password reset tokens; a Bitcoin wallet whose private keys were generated by a buggy `Random()` and were brute-forced from the public key.

**The hard rule, verbatim for any auditor:**

> When generating a token, session ID, password reset token, CSRF token, API key, OAuth state, cryptographic nonce, salt, IV, password salt, JWT secret, or any other value whose unpredictability is part of its security model: use the language's cryptographically secure PRNG. Never `Math.random()`, `random.random()`, `rand()`, `lrand48`, `mt_rand`, `Random()`, `java.util.Random`, `System.nanoTime()` modulus, or anything else seeded from user-observable state.

## Safe primitives per language

| Language / runtime | Use                                                      | Notes                                                       |
|---|---|---|
| Node.js            | `crypto.randomBytes(n)`, `crypto.randomUUID()`, `crypto.getRandomValues(buf)` | All cryptographically secure. Avoid `Math.random()`.         |
| Browser            | `crypto.getRandomValues(buf)`, `crypto.randomUUID()`     | Standard WebCrypto. `Math.random()` is **not** secure.       |
| Python (stdlib)    | `secrets.token_bytes(n)`, `secrets.token_urlsafe(n)`, `secrets.token_hex(n)`, `os.urandom(n)`, `secrets.choice` | `random` is a Mersenne-Twister, **not** secure. `secrets` is the canonical module added in 3.6 for this purpose. |
| Go (stdlib)        | `crypto/rand` — `rand.Read(buf)`, `rand.Int(rand.Reader, max)` | `math/rand` is **not** secure. Always import `crypto/rand`.  |
| Java               | `java.security.SecureRandom` — `nextBytes(buf)`, `new SecureRandom()` | `java.util.Random` is **not** secure. `SecureRandom.getInstanceStrong()` is also fine. |
| Kotlin             | `SecureRandom` (interop with Java)                       | `kotlin.random.Random` is **not** secure.                    |
| Ruby               | `SecureRandom.hex(n)`, `SecureRandom.urlsafe_base64(n)`, `SecureRandom.bytes(n)`, `SecureRandom.uuid` | `Random.new` / `rand` are **not** secure.                    |
| Rust               | `rand::rngs::OsRng` (via `rand` crate), `getrandom::getrandom(buf)` | `rand::thread_rng()` is **not** guaranteed cryptographic — explicit `OsRng` is safer. |
| C / C++            | `getrandom(buf, len, 0)` (Linux), `arc4random_buf(buf, len)` (BSD/macOS), `BCryptGenRandom` (Windows), `RAND_bytes` (OpenSSL) | `rand()`, `random()`, `srand()` are **not** secure. Don't use `time(NULL)` as a seed. |
| .NET               | `RandomNumberGenerator.Create()` → `GetBytes(buf)`, `RandomNumberGenerator.GetBytes(buf)` (.NET 6+), `RandomNumberGenerator.GetInt32(min,max)` | `System.Random` is **not** secure (predictable from a few outputs). |
| PHP                | `random_bytes(n)`, `random_int(min, max)` (≥ 7.0)        | `mt_rand`, `rand`, `uniqid` are **not** secure.              |
| Elixir / Erlang    | `:crypto.strong_rand_bytes(n)`                           | `:rand.uniform` is **not** secure.                           |

## Unsafe patterns to reject

```javascript
// JS / TS — all unsafe for tokens / IDs / keys
const token = Math.random().toString(36).slice(2);
const sessionId = Math.floor(Math.random() * 1e9);
const apiKey = Date.now().toString(36) + Math.random().toString(36);
```

```python
# Python — all unsafe
import random
token = ''.join(random.choices(string.ascii_letters, k=32))
session_id = random.randint(0, 2**31)
csrf = random.random()
```

```go
// Go — math/rand is the wrong import for security-sensitive code
import "math/rand"
token := fmt.Sprintf("%x", rand.Int63())     // unsafe
rand.Seed(time.Now().UnixNano())              // unsafe seed
```

```java
// Java — java.util.Random is unsafe
java.util.Random r = new java.util.Random();
byte[] token = new byte[16];
r.nextBytes(token);                          // unsafe — Random is predictable
long id = System.nanoTime();                 // unsafe — observable
```

```ruby
# Ruby — rand and Random.new are unsafe
token = (0...32).map { rand(36).to_s(36) }.join
id = Random.new.rand(2**31)
```

```c
/* C — rand() / random() / srand(time) is unsafe */
srand(time(NULL));
char token[16]; for (int i=0; i<16; i++) token[i] = rand() & 0xff;
```

```php
// PHP — mt_rand / rand / uniqid are unsafe for security
$token = bin2hex(uniqid());
$id = mt_rand(0, PHP_INT_MAX);
```

## Triager guidance

| What you see | Verdict |
|---|---|
| Non-cryptographic RNG (`Math.random`, `random.random`, `rand`, `mt_rand`, `java.util.Random`, `System.Random`, `Random.new`, `math/rand`, `kotlin.random.Random`, `:rand`) used to produce a token / session / CSRF / API key / nonce / salt / IV / reset code | `confirmed` — the impact is whatever the unpredictable value was supposed to gate. Token-style values: account takeover. Crypto IV reuse: keystream recovery. |
| `time()` / `Date.now()` / `System.nanoTime()` used as the value or as a `srand` seed for a security purpose | `confirmed` — observable to the attacker. |
| Code uses the language's CSPRNG (`crypto.randomBytes`, `secrets.token_bytes`, `crypto/rand`, `SecureRandom`, `random_bytes`, etc.) | `false_positive` — the scanner may have flagged this from naming similarity. |
| Non-crypto RNG used for **non-security** purposes (sampling, shuffling display order, choosing a random colour for a chart, jitter for retries) | `false_positive` — `Math.random` is fine when unpredictability is not in the threat model. Note in your reasoning. |

## Fixer guidance

- Replace the call site with the language's CSPRNG primitive from the table above. Do **not** keep a shim wrapper that picks between secure and insecure based on a feature flag — every call must be secure unconditionally.
- For string tokens, prefer the language's "URL-safe random string" helper if one exists (`secrets.token_urlsafe`, `crypto.randomUUID`, `SecureRandom.urlsafe_base64`) over manually base64-encoding raw bytes; less rope.
- Salts and password hashes: use the password-hashing library's built-in salt generation (bcrypt / argon2 / scrypt all do this). Don't supply your own salt unless you specifically know why.
- Sandbox-checking RNGs: `secrets.SystemRandom()` in Python, `java.security.SecureRandom`, `crypto.randomBytes` in Node — these all read from the OS CSPRNG and are appropriate. Beware code that says "secure random" in a comment but calls `Math.random()` — read the actual call.

## Reviewer guidance

- **Reject** any diff that introduces or retains a non-CSPRNG call for a security-sensitive identifier. Variable name heuristics (`token`, `secret`, `csrf`, `nonce`, `salt`, `iv`, `key`, `cookie`, `session`, `state`, `code`, `reset`) plus the call site's purpose are the signal.
- **Approve** when the safe primitive is used and the output length is appropriate for the purpose (≥ 16 bytes / 128 bits for tokens).
- A non-CSPRNG used clearly outside the security model — e.g. choosing a chart colour, picking a random emoji — is fine. Use judgment on call-site purpose.

## Programmatic guard

`guard_no_insecure_random` in `scripts/lib/guards.sh` flags any diff that adds a non-CSPRNG call (`Math.random`, `random.random`, `random.randint`, `random.choice`/`choices`, `rand`, `mt_rand`, `Random()`, `kotlin.random.Random`, `math/rand`, `java.util.Random`, `System.Random`, `Random.new`, `:rand`, `srand`, `lrand48`) on a line that also contains a credential-shaped identifier name. Pattern is heuristic; if you legitimately need non-crypto randomness on a credential-named variable (rare), restructure or document and the rule will still flag — accept the friction or rename the variable.
